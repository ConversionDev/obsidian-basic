- 구조
	- feature(개인) <— > develop(합치기) <—> main(최종, 배포) 형태
	- hotfix
		- 무언가 에러가 났을 경우 여기에 올리면 trouble shooter가 대신 잡아줌
	- release
		- main 의 업데이트 전 백업 역할 - 1.0.1
		- develop을 main으로 올리기 전 거치는 역할을 할 수 있음

- 전략 반영된 코드를 내 브렌치에 올릴 때
	- 처음이라 깃 클론 하고 ( . 이 없으면 폴더링으로 들어가니까 . 넣기)
		git clone URL (HTTP) .
	- 처음에 clone 하면 main 로 연결되므로, 처음에만 git fetch 실행, 후에 연결하고싶은 브런치 이름으로 체크아웃 하기
		==git fetch==
		git checkout feature-ys
	- (현재 브렌치 알고싶을 때)
		git branch --show-current
	- 내 브랜치(feature-ys)에서 계속 작업
	- 작업 끝날 때
		git add .
		git commit -m "제목"
		git push
	- PR 보내기

- PR 이란?
	- Pull Request(풀 리퀘스트)
	- 내가 만든 코드를 팀의 공식 브랜치(dev ~~또는 main~~)에 합쳐줄 수 있을까?” 라고 요청하는 절차
	- PR을 만들면 GitHub가 다음을 보여줌:
		- 변경된 파일
		- 코드 줄 단위 diff
		- 커밋 목록
		- 리뷰 코멘트
	

- PR 만들기 (우리는 안 씀)
	- GitHub → **Pull Requests** 이동
	- **New Pull Request** 클릭
	- 리스트
		- **base:** `dev`
	    - **compare:** `feature-kjun`
	- PR 제목 및 설명 작성
	- 팀장에게 리뷰 요청
	- 제출
