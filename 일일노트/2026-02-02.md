- LangGraph 쪽 관례 : 튜토리얼/문서에서는 보통 AgentState 를 쓰는 경우가 많다.
- -> ChatState : 역할을 명확히 하기 위해서 변경
- 랭그래프 : 다중 에이전트 사용, 상태 관리
- 바꿀수 있으면 프레임워크가 있어야 파일이 생김 
- 내부 구조 바꿀 수 없으면 플랫폼
- LangGraph는 Google의 Pregel 시스템에서 영감을 받은 메시지 전달(Message Passing)
- 상태 : TypedDict을 상속 받은 클래스
- TypedDict : Python 타입 시스템의 일부고, LangGraph가 “아주 잘 활용하는 도구”다.(정책 기반)
- 규칙 : Base table, 정책 : embedding_table
- 리듀서= 상태 업데이트 로직을 정의하는 핵심 메커니즘=노드가 반환하는 업데이트를 기존 상태에 어떻게 적용할지 결정하는 함수
- 리듀서는 랭그래프에서 메커니즘으로 작동한다
- 들어가는 입구는 넓고 출력하는 곳은 좁아서 양이 줄어들어서 리듀서
- = attension(주목하다 - 욜로할때 얼굴에 사각형 그리는거, 불필요한 정보 버리고 하나에 집중 하는 거)
- 엑사원은 기억하는 능력은 없지만 메커니즘(통찰)을 가지고 있다.
- 훅(리액트 함수) : 배워야 하는것 - 값을 바꾼다
- 매커니즘 함수 : 내장된 함수 외부에서 주어지지 않고도 타고난 기능 add
- 상태객체 내부 : 아무거도 없으면 replace, 
- 새로 하는 질문 : 질문이 이어서 가게 하고 싶으면 reduce add 추가
- 노드 : 랭그래프 작업 단위, 함수(데이터 전달자) - transfer 상태(typed의 자식) 함수 
- - 값을 바꾼다
- 구현해야 할것(상태를 파라미터로 받음 - 노드의 파라미터는 상태다 - 상태 전달하는게 노드다)
- 리듀서 : 메커니즘(내장된 함수)
- 내장된 객체(임베디드, 빌트인)
- 맨 처음 객체 : 그래프
- DB도 동기
- 베이스 테이블에 데이터를 랭그래프를 이용해서, 임베딩 테이블로 이동시킬때, 넥스트에서 트리거를 작동시킬 때, Batch 작업을 비동기 작업의 큐방식으로 백그라운드에서 돌리려고해.
- 올려 놓고 안내리는 거 : 무중단 서비스 올려둔 상태로 그냥 작업하는거.
